ACL17-Get To The Point: Summarization with Pointer-Generator Networks

PDF, Blog

这是我本周要讲的论文，发表在17年ACL会议上。本文针对长文本生成式摘要中出现的上下文不连贯、事实错误以及未登录词的问题，提出了Pointer-Generator模型，将复制与生成相结合，同时针对重复生成句子的问题，提出了覆盖机制。

Background

  • Extractive methods： assemble summaries exclusively from passages
  • Abstractive methods: generate novel words and phrases not featured in the source text

Motivation

Two shortcomings(multi-sentence summaries):

  • reproduce factual details inaccurately and inability to deal with out-of-vocabulary
  • tend to repeat themselves

Two solution:

  • copying words from the source text via pointing
  • coverage vector

Model

Experiment Tasks and Datasets

  • [Experiment Tasks and Datasets]
    • Reading Comprehension (bAbI)
    • Sentiment Classification (Stanford Sentiment TreeBank)
    • Sequence Modeling - POS tagging (WSJ-PTB)

  • [Model]

    • Input Module
    • Question Module
    • Episodic Memory Module
    • Answer Module

  • [Dynamic Memory Networks V.S. Memory Networks]

    1. Sentence Encoding: RNN, Position-Encoding
    2. Dynamic: Episodic Memory Module(When attention probbility is in the position), Multi-Hops
  • [Motivation for Memory Module]:

    Episodic Memory Module: Given a collection of input representations, the episodic memory module chooses which parts of the inputs to focus on through the attention mechanism. It then produces a ”memory” vector representation taking into account the question as well as the previous memory. Each iteration provides the module with newly relevant information about the input. In other words, the module has the ability to retrieve new information, in the form of input representations, which were thought to be irrelevant in previous iterations.

  • [Motivation for Dynamic Part]:

    Need for Multiple Episodes: The iterative nature of this module allows it to attend to different inputs during each pass. It also allows for a type of transitive inference, since the first pass may uncover the need to retrieve additional facts. For instance, in the example in Fig. 3, we are asked Where is the football? In the first iteration, the model ought attend to sentence 7 (John put down the football.), as the question asks about the football. Only once the model sees that John is relevant can it reason that the second iteration should retrieve where John was. Similarly, a second pass may help for sentiment analysis as we show in the experiments section below.

网络爬虫之正则表达式

在了解了如何通过httpclient获取到相关网页的源代码后，作为主题爬虫，还需要获取网页源代码中的相关信息以及下一跳的URL地址。这就需要用到正则表达式，这里总结一下处理网页源代码时用到的一些技巧。

  • 分组
    在我们获取网页中关键信息时，我们或许会使用多次正则，而如何获取我们所指定位置的正则所匹配的信息，我们就需要对正则表达式进行分组。
    比如

我们要获取chapterName处的内容，就是group(1)，wordNum处的内容，就是group(2)，updateTime处的内容，就是group(3)。第0组group(0)或者不写值group()即是匹配整个字符串，从左往右起第一个左括号即是分组1，以此类推。

  • 贪婪模式
    贪婪模式即是匹配尽可能多的字符串，比如.*或者.+,同样上面例子中的chapterName=\”(.+)\”，将会匹配从chapterName=”开始，直到找到最后一个”才匹配结束。所以我们在处理网页时更常用的是懒惰模式。

  • 懒惰模式
    懒惰模式是在能使整个匹配成功的前提下使用最少的重复。比如.*？或者.+？，同样上面例子中的chapterName=\”(.+？)\”，将会匹配从chapterName=”开始，找到第一个”就匹配结束。

中文分词原理之最大长度匹配法

发表于 2017-01-14 | 分类于 知识小结

最大长度匹配法

最常见的分词方法为查找词典算法，其中最常用就是最大长度匹配法——从字符串的指定位置开始，匹配词典，找出最长词。

正向最大长度匹配法

人们用句的时候往往更倾向与使用更短的词，只有在必要的情况下才会用长词来表述，比如大学和大学生，如果大学能够表示大学生，那么人们更倾向与用短词，既然长词出现，那么就肯定是有含义的，需要优先匹配。这便是最大长度匹配法。最大长度匹配法常用前缀树（标准Trie树)进行匹配。

逆向最大长度匹配法

我们会发现正向最大长度匹配的精度不高，会出现歧义，比如大学生活丰富多彩，切分结果为大学生/活/丰富多彩，而采用逆向最大长度匹配的结果为大学/生活/丰富多彩，可见逆向最大长度匹配可以消除部分歧义，精度稍高。逆向最大长度匹配常用后缀树进行匹配。

标准Trie树（前缀树）

前缀树的搜索速度快在于其每一个节点都指向一个直接定址表，无需处理冲突，而且树的高度小，所以 搜索 速度很快。但是它是空间换时间的算法，以内存消耗为代价。如果只是26个英文字母，每个节点就要保存26个指针，但并不是每个指针数组都保存记录，空的指针数组导致内存空间的浪费。

因而我们可以结合了Trie树时间效率的优势和二叉搜索树空间效率的三叉搜索树。

三叉搜索树

三叉搜索树使用了一种聪明的手段去解决Trie的内存问题（空的指针数组）。为了避免多余的指针占用内存，三叉搜索树的节点表示成“树中有树”的形式。与Trie树不同的是，三叉搜索树的结构与读入顺序有关，三叉树是否平衡取决于其读入顺序。我们可以把词典按词的首字母大小排序，每次取中间值读入来构建平衡树，这样在进行正向最大长度匹配时，从一句话的指定位置开始，如果将该字首字母与根节点的字的首字母匹配，如果相等则继续访问中间孩子节点，如果小于则比较左孩子节点，如果大于则继续比较右孩子节点。匹配完一个词则回到根节点继续匹配。直到匹配到该句结尾，如果没有匹配的词则按单字切分。

今朝龙翔微信社区技术总结

这几天突然接到了之前接受的今朝龙翔社区的项目即将上线的消息，于是开始抓紧时间赶紧进度，同时在需求方面又发生了很大的变动，原本是以将webAPP融入微信的方式（通过菜单链接直接链到webAPP中）,现在需要改成将webAPP的功能直接拆分到微信端，通过微信菜单直接获取资讯图文或者链接到单页面，所以在此基础上为了后期可拓展性，我并没有把原来的webAPP之前拆了，而是在此基础上进行了功能的拓展。下面是新旧版本的对比：


之后经过这几天的赶工，项目总算是完工了，现总结一下这次项目的一些心得。

一、微信接口获取用户信息

微信获取用户信息可以通过两种方式，两种方式用到的接口也不一样。

###（1）通过自定义菜单的click类型获取用户信息（无需高级接口）
如果是通过自定义菜单的click类型，以图文链接的方式获取用户信息，可以先通过FromUserName获取到用户的openID，之后即可通过获取用户信息的接口https://api.weixin.qq.com/cgi-bin/user/info?access_token=ACCESS_TOKEN&openid=OPENID&lang=zh_CN 获取用户基本信息。

###（2）通过自定义菜单的view类型获取用户信息（需要高级接口，即认证服务号，另外还需要域名）
如果是通过自定义菜单的view类型，以自定义菜单直接链接到获取用户信息的页面，菜单的url是一个授权回调url，这样才能在菜单点击的时候使用oauth2.0授权获得openid，接口如下https://open.weixin.qq.com/connect/oauth2/authorize?appid=APPID&redirect_uri=REDIRECT_URI&response_type=code&scope=SCOPE&state=STATE#wechat_redirect 。其中的redirect_uri便是回调页面的链接，不过页面所在的域名必须在微信公众号的后台进行绑定授权，授权后可以回调该域名下面的页面；其中的scope参数分为snsapi_base和snsapi_userinfo两种：

1、以snsapi_base为scope发起的网页授权，是用来获取进入页面的用户的openid的，并且是静默授权并自动跳转到回调页的。用户感知的就是直接进入了回调页（往往是业务页面）

2、以snsapi_userinfo为scope发起的网页授权，是用来获取用户的基本信息的。但这种授权需要用户手动同意，并且由于用户同意过，所以无须关注，就可在授权后获取该用户的基本信息。

###（3）两种方式总结：

用户管理类接口中的“获取用户基本信息接口”，是在用户和公众号产生消息交互或关注后事件推送后，才能根据用户OpenID来获取用户基本信息。这个接口，包括其他微信接口，都是需要该用户（即openid）关注了公众号后，才能调用成功的。

二、多图文上传

原本是打算找一个合适插件来做随手拍的功能的，但找来找去并没找到符合要求的插件，接口那边需要通过POST文件的方式，而我发现大多数插件都是通过ajax并发上传单文件的方式，所以为此很头疼，所以最终索性自己来实现这个功能，用HTML5的File API来做多图上传及预览的随手拍功能。

三、微信APP下载

最后一个需要拿出来特别提一下的就是微信中实现app下载的功能，这是一个比较坑的地方，微信公众号拦截APPStore跳转和APK下载，允许使用腾讯域名下面的下载链接，所以解决方法有两个：

  • 1 把应用拿到应用宝，换得应用宝的APP下载地址，放到微信里就能跳转了
  • 2 做一个漂亮点的下载页面，并提示用户在浏览器中打开
